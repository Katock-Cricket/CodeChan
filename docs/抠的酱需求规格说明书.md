# 抠的酱（CodeChan）需求规格说明书

## 1. 引言

### 1.1 编写目的

本文档用于明确“抠的酱（CodeChan）”项目的功能需求、非功能需求和验收边界，为后续系统设计、编码实现、测试验收与迭代拆分提供基准依据。

### 1.2 产品范围

本产品是一款基于 VS Code / Cursor 等 VS Code 兼容 IDE 的代码辅助工具。它通过在 IDE 内嵌 3D 二次元角色（名为“抠的酱”），结合本地音频模型、云端 LLM 与定制化 OpenCode Fork，为开发者提供具备情绪价值的语音交互、可视化 Agent 工作流及智能编程辅助。

### 1.3 核心原则

- **本地音频优先且仅本地音频：** ASR 与 TTS 均在本地后端执行，不向云端上传用户语音、合成文本对应的音频请求或本地音色模型。
- **云端仅用于 LLM：** 闲聊、意图路由、情境化回复和 Agent 指令整理可调用云端 LLM，但需遵循用户配置的供应商、Base URL 和密钥来源。
- **高质量 Agent 事件优先：** OpenCode 集成坚持使用定制 Fork，以获取细粒度、高质量、结构化事件，满足角色动作、IDE 联动和任务上下文表达的需要。
- **可替换模型架构：** 首发默认使用 IndexTTS-2、Faster-Whisper 和 OpenCode Fork，但系统必须预留可替换 Adapter，以便后续切换音频模型、LLM 供应商或 Agent 实现。

## 2. 功能模块划分

系统按工程职责划分为六个一级模块：

1. **VS Code Extension Shell：** 负责插件激活、命令注册、侧边栏、Webview 容器、设置页、SecretStorage、与 IDE API 的桥接。
2. **Avatar Runtime：** 负责 VRM 加载、Three.js 渲染、动作栈、表情、Lip-sync、LookAt、道具命中检测和动画资源管理。
3. **Local Server：** 负责 WebSocket 服务、ASR、TTS、LLM 调用编排、意图路由、任务上下文管理和本地运行状态监控。
4. **Agent Adapter：** 负责 OpenCode Fork 的启动、终止、状态监听、事件标准化、任务注入、异常恢复、日志与并发任务管理。
5. **Protocol Layer：** 负责前后端统一消息协议、错误码、事件版本、二进制音频帧约定和兼容性策略。
6. **Config Layer：** 负责角色人设、VRM 模型、音色模型、道具配置、动作映射、LLM 凭据来源和降级策略。

## 3. 功能需求

### 3.1 伴侣渲染与表现模块

本模块负责 3D 角色在 VS Code Webview 中的视觉呈现和动作表现。

- **[REQ-REN-01] 3D 模型加载与渲染：** 系统必须能够在 Webview 中加载并渲染标准 VRM 格式 3D 模型，支持材质、贴图、Humanoid 骨骼、BlendShape / Expression、Spring Bone 等 VRM 常用能力。
- **[REQ-REN-02] 模型资源校验：** 用户导入 VRM 模型时，系统需检查文件存在性、扩展名、基础 VRM 元数据、文件大小上限和加载失败原因，并在 UI 中给出明确提示。
- **[REQ-REN-03] 待机动画系统：** 系统应内置呼吸、轻微晃动、随机眨眼等待机动画，并在无交互状态下循环播放。
- **[REQ-REN-04] 面部表情控制：** 支持通过 API 指令触发角色表情，如开心、疑惑、思考、抱歉、紧张、害羞等。表情强度应可被配置中的情绪表达系数影响。
- **[REQ-REN-05] 唇形同步：** 角色在播放 TTS 语音流时，需根据音频振幅、音素或 TTS 侧输出的口型帧驱动嘴部形变，实现基本口型匹配。
- **[REQ-REN-06] 动作播放与动作栈调度：** 角色能够播放预设动画，并通过四级优先级动作栈进行调度管理。

| 优先级 | 级别名称 | 典型动作 | 调度行为 |
|--------|----------|----------|----------|
| **P0** | 即时交互级 | 摸头反应、被道具击中、扭身躲闪 | 立即打断 P1/P2/P3；播放完毕后恢复栈中次高优先级动作 |
| **P1** | 情绪对话级 | 说话手势、开心点头、情绪化反应 | 打断 P2/P3；通常与 TTS 语音等长；结束后恢复 P2/P3 |
| **P2** | 任务执行级 | 前倾看文件、专注打字、托腮思考 | 打断 P3；在 Agent 工作状态持续期间循环；Agent idle 后恢复 P3 |
| **P3** | 待机循环级 | 呼吸起伏、轻微晃动、随机眨眼 | 最低优先级，作为兜底动作持续循环 |

调度规则：同一级别内只保留最新动作；高优先级动作结束后检查栈并恢复最高级别的未完成动作；P0/P1 为事件型，P2 为状态型，P3 为持续型。

### 3.2 语音与自然交互模块

本模块处理用户与 3D 伴侣之间的语音、文字和角色互动。

- **[REQ-INT-01] 语音录制入口：** 提供 UI 麦克风按钮和 VS Code 内部快捷键控制语音录制。快捷键仅在 VS Code / Cursor 窗口内生效，不提供系统级全局热键。
- **[REQ-INT-02] 语音转文本：** 本地后端需接收前端音频流，并调用本地 ASR 模型（默认 Faster-Whisper）转为文本。系统不对 ASR 准确率设定业务验收目标，识别质量主要由用户选择的模型、语言和硬件条件决定。
- **[REQ-INT-03] 意图路由：** 系统需判断用户输入文本的意图，并路由到“即时互动”“Agent 任务”“动作指令”“闲聊回复”中的一种或多种处理流。
  - **路由优先级：** 显式 UI 事件优先级最高；显式 Agent 模式输入次之；明确动作/道具/身体互动指令再次；包含代码任务、文件操作、命令执行、项目分析等语义时进入 Agent 任务；其余进入闲聊。
  - **混合意图处理：** 当输入同时包含闲聊与代码任务时，优先创建 Agent 任务，并允许生成一条短的伴侣确认回复；当输入同时包含动作指令与闲聊时，先执行动作，再生成闲聊回复；当用户在 CodeChan 侧边栏中手动选择 `chat` 或 `agent` 模式时，以用户显式模式为准。
  - **低置信度处理：** 当路由置信度不足时，默认进入闲聊，并可用简短追问确认是否要下发 Agent 任务，避免误触发代码修改或命令执行。
- **[REQ-INT-04] 闲聊与情绪回复：** 对于非 Agent 任务，调用云端 LLM 生成符合角色人设的回复文本，并返回情绪标签、表情强度和可选动作建议。
- **[REQ-INT-05] 文本转语音：** 本地后端需将回复文本通过本地 TTS 模型合成为角色音色音频流。首发默认使用 IndexTTS-2，并通过 TTS Adapter 预留后续切换 VITS、Fish Speech 或其他本地 TTS 模型的能力。
- **[REQ-INT-06] 简单互动响应：** 当用户在 Webview 中点击角色特定区域（如头部、身体、手部）时，需触发对应 P0 动画和预置语音反馈。
- **[REQ-INT-07] 复杂互动响应：** 用户可以使用道具（蛋糕、皮鞭等）对角色进行嘉奖或惩罚，也可通过言语进行嘉奖或惩罚。
  - **道具 UI 形态：** CodeChan 插件侧边栏内设置独立的道具子面板，展示预置道具列表（图标 + 名称）。用户点击道具后，Webview 内光标切换为对应道具图标；随后点击角色模型触发互动。点击 Webview 外区域或按 Esc 键取消道具选择。
  - **即时响应机制：** 道具/言语互动触发后，角色立即播放对应 P0 动画及预置语音。P0 动作和预置语音通常设计为 3-5 秒，作为即时反馈。
  - **情境化后续回复：** 触发 P0 的同时，后端并发调用云端 LLM，结合当前 Agent 工作流上下文生成情境化回复，并调用本地 TTS 合成 P1 对话语音。即使 P1 内容提前准备完成，也必须等待 P0 播放结束后再进入 P1，避免即时反馈被后续语音覆盖。

### 3.3 代码 Agent 可视化联动模块

本模块负责将底层代码 Agent 的不可见行为转化为 IDE 行为和角色表演。

- **[REQ-AGT-01] Agent 进程管理：** 本地后端需具备启动、终止、重启、健康检查和监控 OpenCode Fork 进程的能力。需记录进程 PID、工作目录、启动参数、任务状态、最近心跳、退出码和异常原因。
- **[REQ-AGT-02] 状态拦截与解析：** 系统需通过 OpenCode Fork 内置 Hook 或事件总线获取结构化 Agent 事件，如 `thinking`、`reading`、`writing`、`executing`、`waiting_user`、`error`、`completed`。不以标准输出文本解析作为主路径。
- **[REQ-AGT-03] 行为-动作映射：** 建立可配置映射表，将 Agent 状态映射为角色动作、表情、LookAt 和可选语音提示。
  - 示例：状态 `reading`，上下文 `README.md` -> 动作 `lean_forward_and_read`，表情 `focused`，视线锁定编辑器区域。
- **[REQ-AGT-04] IDE UI 联动控制：** Agent 查阅文件时，VS Code 自动打开对应文档，并在事件提供范围信息时滚动到 Agent 具体查看的范围。不做高亮标记，不做额外 Diff / 预览渲染。
  - 若该文档由 CodeChan 为可视化临时打开，且用户在此期间没有主动编辑、固定、切换到该文档或显式保留，则在对应可视化事件结束后自动关闭该文档，避免占用编辑器空间和内存。
  - Agent 写代码或执行命令时，编辑器侧不做额外渲染，但角色需依据映射表执行 P2 级任务动画。
- **[REQ-AGT-05] 双轨任务下发：** 用户可以通过 CodeChan 自有侧边栏输入 Agent 指令，也可以通过 3D 伴侣语音/文字下达编程任务；底层统一注入 OpenCode Fork。OpenCode 自身的原生终端输入继续保留，但不纳入 CodeChan 侧边栏定义。
- **[REQ-AGT-06] 并发任务策略：** 首发版本同一工作区仅允许一个活跃 Agent 任务。新任务到来时，系统需根据来源和用户选择执行排队、拒绝、取消当前任务后启动新任务三种策略之一；涉及取消当前任务时必须有明确用户确认。

### 3.4 基础系统与配置模块

- **[REQ-SYS-01] 角色详情配置：** 用户可在设置面板切换或导入本地 VRM 模型、加载本地音色模型、自定义角色人设、语言、称呼、回复风格和情绪表达强度。
- **[REQ-SYS-02] 模型自定义导入：** 用户导入模型时，系统需支持选择本地路径、保存引用、校验可访问性、记录模型元数据、测试加载结果和恢复默认配置。
  - VRM 模型需校验文件类型、大小上限、基础元数据和渲染加载结果。
  - TTS 音色模型需校验目录结构、必要权重文件、配置文件、采样率/说话人信息和模型 Adapter 兼容性。
  - ASR 模型需允许切换本地模型路径或模型名称，并在首次使用前执行轻量健康检查。
- **[REQ-SYS-03] LLM 密钥管理：** 提供安全 UI 配置云端 LLM 的 API Key、Base URL、模型名和供应商。密钥来源兼容系统环境变量与 VS Code SecretStorage，不提供明文落盘存储。
- **[REQ-SYS-04] 通信状态监控：** 前端 UI 需清晰展示与本地 Python 服务端的 WebSocket 连接状态，如在线、断线、重连中、后端未启动、模型加载中。
- **[REQ-SYS-05] 日志与诊断：** 系统需提供可控日志输出，覆盖插件、Webview、本地后端、Agent Adapter、ASR/TTS、LLM 调用和协议错误。日志不得泄露 API Key、用户完整语音数据或敏感环境变量。

## 4. 非功能需求

### 4.1 性能需求

- **渲染性能：** 在推荐硬件上，Webview 中 3D 渲染应以 30 FPS 以上为可用目标，轻负载场景力争接近 60 FPS。系统需提供降低渲染质量、关闭部分物理效果或降低动画频率的选项。
- **文字交互延迟：** 纯文字交互的端到端耗时主要受云端 LLM 影响。系统应支持流式文本展示，在云端 API 正常响应时尽快显示首段回复，不以固定秒数作为硬性验收。
- **语音交互延迟：** 语音交互需支持分阶段反馈：录音结束后显示“识别中/思考中/合成中”状态；在本地模型和云端 LLM 正常响应时，应尽量缩短 ASR、LLM、TTS 的首包延迟，不设绝对 4 秒硬门槛。
- **复杂互动延迟：** 道具或言语互动必须先通过本地预置 P0 动作和语音完成即时反馈，随后在后台准备 P1 情境化回复。P1 的可播放时间受云端 LLM 和本地 TTS 影响，不要求在 P0 内必然完成。
- **资源占用：** 本地后端需支持按需加载 ASR/TTS 模型、空闲释放或用户配置资源上限。显存、内存目标以模型规格和用户硬件为准，不设置单一固定上限。

### 4.2 可用性需求

- **安装与初始化：** 提供一键安装包或详细环境配置脚本（如 `setup.bat` / `setup.sh`），自动检查 Python 运行环境、CUDA / MPS / CPU 后端、模型目录和必要依赖。
- **可见降级：** 若初始化时本地硬件不足、模型缺失或 ASR/TTS 超时，系统可降级为文字显示、静音回复或关闭语音输入，但需在 UI 中明确展示当前降级状态，不应静默失败。
- **错误恢复：** 后端断连、Agent 崩溃、模型加载失败、LLM 超时等情况应给出可理解提示，并提供重连、重试、重新加载模型或查看日志入口。

### 4.3 兼容性需求

- **操作系统：** 核心功能目标支持 Windows 10/11、macOS（Intel 与 Apple Silicon）及主流 Linux 发行版（优先 Ubuntu）。
- **IDE 版本：** 兼容 VS Code v1.80 及以上版本，并优先适配 Cursor 的 VS Code 扩展运行环境。
- **音频模型兼容：** 首发保证默认模型链路可运行；第三方本地模型以 Adapter 能力和用户本地环境为准。

### 4.4 安全与隐私需求

- **本地音频处理：** ASR 与 TTS 必须在本地执行，不向云端音频服务上传用户语音或合成请求。
- **云端 LLM 边界：** 云端仅用于 LLM 文本推理。发送给 LLM 的上下文应遵循最小必要原则，并在设置中允许用户控制是否携带 Agent 工作流上下文、当前文件摘要或完整片段。
- **凭据存储：** LLM API Key 必须从系统环境变量或 VS Code SecretStorage 读取。配置文件中只允许保存供应商、Base URL、模型名、环境变量名或 SecretStorage 引用标识，不保存明文密钥。
- **日志脱敏：** 日志输出需脱敏 API Key、Authorization Header、用户路径中的敏感片段和模型配置中的秘密字段。

### 4.5 可扩展性需求

- **Agent 接口标准化：** 系统对 OpenCode Fork 的调度需封装 Adapter 接口。首发坚持 OpenCode Fork，但接口设计需允许未来接入其他 Agent。
- **模型接口标准化：** ASR、TTS、LLM 均需通过 Adapter 或 Provider 接口接入，避免业务逻辑绑定具体模型 SDK。
- **多语言支持：** 系统架构需预留 i18n 国际化接口，首发支持中文和英文交互。
- **协议版本化：** WebSocket 消息协议需包含版本或能力协商机制，以便前后端独立升级时保持兼容。
